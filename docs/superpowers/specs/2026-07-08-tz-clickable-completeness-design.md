# Дизайн: Полная кликабельность по ТЗ (React PWA)

> Дата: 2026-07-08 · Автор: LLM-сессия (brainstorming) · Статус: **на утверждении**
>
> Основано на аудите существующего React PWA против 18-пунктового чек-листа ТЗ
> (workflow `zovu-tz-audit`, 6 агентов, 28 находок с `file:line`). Решение по стеку:
> **применяем чек-лист ТЗ как аудит полноты к существующему React PWA** (не Flutter —
> см. [ADR-008](../../09-decisions.md)). Ничего готового не выбрасываем.

## 1. Цель

Довести React PWA до состояния, которое требует ТЗ прототипа:

- **Каждая кнопка** либо ведёт на реальный следующий экран, либо меняет реальное
  состояние. Ноль мёртвых кнопок и тупиков навигации.
- **Каждый список** имеет пустое состояние; **каждое async-действие** — loading;
  сетевой сбой показывает **error + retry**, а не маскируется под «пусто».
- Оба сквозных флоу (заказчик и специалист) проходятся от начала до конца без
  ошибок в консоли.

Бэкенд и стек остаются. Данные — реальные (NestJS+Prisma), не моки.

## 2. Что уже готово — НЕ трогаем

Подтверждено аудитом с `file:line`:

- Онбординг Welcome→телефон(+7)→OTP(таймер/авто-submit/dev-1111), dev-быстрый вход,
  guards роутов, выбор роли + анкеты заказчика/специалиста (кроме диплома).
- **Сквозной happy-path обеих ролей** (п.4/5/7): карта/лента → отклик → принять/
  отклонить (каскад «Не выбран» + чат + комиссия) → чат (WS) → завершение → оценка.
- **п.13 Деньги целиком** (M5): баланс → пополнение → блок «на нуле» (403
  `subscription_inactive`) → разблокировка. Целостно.
- **п.9** Пустые состояния «0 откликов» (MatchingState) и «0 заказов в радиусе».
- В ленте специалиста мёртвых кнопок нет.

## 3. Зафиксированные развилки

1. **Контрпредложение (п.6/11)** → **лёгкий реальный round-trip**. Одна миграция
   Prisma: `Bid.counterPrice Int?` + значение `countered` в enum `BidStatus`.
   Не отдельная модель истории торга.
2. **Разрешения (п.1)** → **primer-экраны + локальные разрешения**. Экраны «зачем»
   перед системным диалогом для гео/push/камеры + вызовы
   `navigator.geolocation` / `Notification.requestPermission` / camera. **Без**
   серверного web-push (VAPID/подписка/SW-push вне скоупа этого прохода).

## 4. Майлстоуны

Идём G1→G6, майлстоун за майлстоуном. **Гейт каждого:** `apps/web` собирается
(`npm run build`, 0 ошибок) + `apps/api` собирается, jest зелёные, обновлён
[10-status.md](../../10-status.md), один conventional-commit, короткий отчёт
пользователю. Не переходим к следующему, пока текущий флоу не кликается целиком.

### G5 — Permission-primers (п.1)

Экраны-объяснения «зачем» ПЕРЕД системным диалогом. Без серверного push.

- Компонент `PermissionPrimer` (иконка, «зачем», кнопки «Разрешить»/«Позже»).
- **Гео**: primer перед первым `navigator.geolocation.getCurrentPosition`
  (сейчас `useGeo.ts:11` дёргает напрямую). Место: после OTP при первом входе
  специалиста и/или перед картой. При «Позже»/отказе — текущий fallback на Алматы
  (уже есть), но осознанный.
- **Push**: primer → `Notification.requestPermission()` (локально, без подписки на
  сервер). Честный текст «включить уведомления на этом устройстве».
- **Камера**: primer перед capture в `VerificationScreen` (селфи/документ).
- Роуты онбординга разрешений между OTP и shell (можно пропускать; идемпотентно —
  не показывать повторно, если разрешение уже дано; флаг в localStorage).
- i18n: новые строки-объяснения (ru+kk).

### G6 — Контрпредложение по цене (п.6/11) — round-trip

**Backend (Prisma миграция `bid_counteroffer`):**
- `Bid.counterPrice Int?` — встречная цена заказчика.
- `enum BidStatus` += `countered`.
- `bids.service.ts`:
  - `counter(userId, bidId, price)` — заказчик шлёт встречную: проверка владельца
    заказа, `status: 'countered'`, `counterPrice: price`, push специалисту
    (`type: 'bid_countered'`).
  - специалист отвечает: **принять встречную** (`accept` использует `counterPrice`
    как финальную цену + пересчёт комиссии) либо **перебить своей** (существующий
    `create`/upsert с новой `price`, сброс `counterPrice`, статус снова `pending`).
- `bids.controller.ts`: `POST /bids/:id/counter` (заказчик).
- jest-тест: round-trip pending→countered→accepted; комиссия считается от финальной
  цены; чужой заказ = 403.

**Frontend:**
- `OrderBidsScreen` — третья кнопка «Предложить свою цену» рядом с Принять/Отклонить
  → `BottomSheet` ввода цены → `counterBid(bidId, price)`.
- Отображение статуса `countered` («Вы предложили X ₸, ждём ответа»).
- Специалист (`MyBidsScreen`/деталь отклика) — карточка встречной цены → «Принять
  X ₸» / «Предложить другую» (переиспользуем `RespondSheet`).
- i18n (ru+kk) + `bidStatus.countered`.

## 7. Критерии готовности (проверка перед «готово»)

Для каждого G — прогон соответствующего флоу вживую (скриншоты через headless
Chrome/CDP, как в прошлых спринтах) + сборка + jest. Финальный DoD (после G6):

- Заказчик: онбординг → primer-разрешения → создать заказ (валидация с inline) →
  получить отклик → **контрпредложение** → принять → чат → завершить → оценить →
  (уметь отменить до принятия; пожаловаться) — целиком без правки кода между шагами.
- Специалист: онбординг → верификация (диплом+камера-primer) → лента (error-состояние
  при сбое) → отклик → **ответ на встречную цену** → принятие → чат → завершить →
  управлять несколькими активными по статусам.
- Общее: чат-тред с поддержкой, многошаговая жалоба, переключение ролей в обе
  стороны, каждый список — loading/empty/error, ноль мёртвых кнопок.

## 8. Вне скоупа

- Серверный web-push (VAPID/подписка/SW) — отдельный этап.
- Полная модель истории торга (BidOffer) — выбран лёгкий round-trip.
- Маркеры специалистов на карте заказчика (нужна персистентная гео специалиста) —
  давний follow-up.
- Платежи (эквайринг) — вне скоупа проекта.
- Экраны Безопасность/Приватность настроек — убраны до появления контента.
