# ZOVU — Master Prompt для Claude Code

> **Как выполнять:** прочитай этот файл целиком. Затем выполняй строго по фазам, начиная с **Phase 0 (LLM Wiki)**. Не пиши код приложения, пока Phase 0 не завершена. Мокапы лежат рядом (папка `design/` или приложены к сообщению) — они канон по структуре экранов и тону, пиксель-в-пиксель не требуется.

---

## 0. Миссия

Собрать **Zovu** — C2C-маркетплейс услуг для Казахстана (аналог Profi/inDrive для бытовых услуг). Заказчик публикует заказ с ценой и гео, специалисты рядом видят его на карте/в ленте/в свайп-колоде и откликаются. Дальше: принятие отклика → чат → выполнение → двустороннее подтверждение → взаимные оценки.

Результат: монорепо с Flutter-приложением (iOS/Android), NestJS-бэкендом, мини-админкой и живой демо-сценой на сид-данных (Алматы).

## 1. Скоуп

### 1.1 In scope
- Две роли на одном аккаунте: **Заказчик** и **Специалист (частный)**, переключение без релогина.
- Auth по номеру телефона + SMS OTP (провайдер за интерфейсом, в dev — мок).
- Верификация специалиста: селфи + селфи с документом; статус **«Дипломированный специалист»** через загрузку диплома.
- Заказы: создание (категория, описание, до 5 фото, бюджет, адрес/гео), лента, карта, **Tinder-колода**, отклики (принять цену / предложить свою), каскадный отказ остальным.
- Монетизация специалиста: баланс, подписка **100 ₸/день**, комиссия за заказ (см. ADR-001), пополнение (Kaspi/карта — мок).
- Чат (текст, WebSocket), push-уведомления (мок + FCM-адаптер).
- Завершение заказа с двусторонним подтверждением и автозакрытием через 24 ч.
- Отзывы 1–5★ + ИИ-фильтр (заглушка за интерфейсом) + жалобы и пост-модерация.
- Служба поддержки (тикеты-чат с категориями и вложениями).
- Фильтры подбора специалистов на заказе.
- Локализация **RU / KZ** (переключение в настройках).
- Мини-админка: очереди верификации, дипломов, пользовательских категорий, жалоб на отзывы, тикетов поддержки.

### 1.2 Out of scope — НЕ ДЕЛАТЬ НИ ПРИ КАКИХ УСЛОВИЯХ
- **Весь B2B-модуль (раздел 12 ТЗ): компании, менеджеры, сотрудники компаний, корпоративная панель, тарифные планы компаний, регистрация через администратора.** Слов «компания», «менеджер», «сотрудник компании» в домене, UI и коде быть не должно.
- Фильтр «Тип исполнителя» (Ф-06) — удалён вместе с B2B.
- Автоперевод сообщений и эмодзи-панель в чате (Ч-03/Ч-04 из ТЗ v1.2) — только plain text.
- Реальные платежи, реальный SMS-шлюз, реальный LLM-модератор — только интерфейсы + dev-моки (прод-адаптеры — заглушки с TODO).
- Web-версия для пользователей (админка — единственный web).

## 2. Стек и структура репозитория

```
zovu/
├── CLAUDE.md               # краткий вход для LLM (создаётся в Phase 0)
├── ZOVU_PROMPT.md          # этот файл
├── docs/                   # LLM Wiki — source of truth (Phase 0)
├── docker-compose.yml      # postgres(+postgis), minio, api
├── apps/
│   ├── mobile/             # Flutter 3.x / Dart 3
│   ├── api/                # NestJS 10 + Prisma + PostgreSQL 15 + PostGIS
│   └── admin/              # Vite + React + TS (минимальный, без дизайн-требований)
└── .env.example
```

**Mobile:** Flutter, Riverpod (hooks_riverpod), GoRouter, dio, freezed + json_serializable, intl/ARB (ru, kk), flutter_map (OSM-тайлы) + geolocator, socket_io_client, flutter_animate, confetti, image_picker + flutter_image_compress (НФ-08). Архитектура feature-first: `lib/features/<feature>/{data,domain,ui}` + `lib/core/{theme,router,network,l10n}`.

**API:** NestJS, Prisma, PostgreSQL + PostGIS (гео-запросы `ST_DWithin`/`ST_Distance`), Socket.IO gateway (чат), `@nestjs/schedule` (кроны), JWT (access 15 мин / refresh 30 дней), class-validator, Swagger (экспорт `openapi.json` в `docs/api/`). Файлы — MinIO (S3-совместимый) локально.

**Admin:** Vite + React + TS + любая лёгкая таблица (например TanStack Table). Логин по статическому админ-токену из `.env`. Дизайн — функциональный, без требований.

Если какая-то библиотека недоступна/конфликтует — замени аналогом и зафиксируй в ADR.

## 3. Phase 0 — LLM Wiki (обязательный первый шаг)

До любого кода создай `docs/` — вики, по которой любая LLM-сессия (и человек) может продолжить проект без потери контекста:

```
docs/
├── 00-overview.md        # что такое Zovu, роли, ключевые сценарии, ссылки на остальные страницы
├── 01-scope.md           # in/out, полный список исключений B2B, дельты от ТЗ (см. Приложение B)
├── 02-architecture.md    # монорепо, стек, схема взаимодействий (mermaid), env-переменные
├── 03-data-model.md      # ERD (mermaid) + пояснения к Prisma-схеме, инварианты
├── 04-api.md             # обзор эндпоинтов + ссылка на openapi.json, WS-события
├── 05-screens.md         # карта экранов S-01…S-35: роут, состояние, переходы, диплинки
├── 06-design-system.md   # токены, компоненты UI-кита, моушн, haptics, правила
├── 07-business-rules.md  # подписка/баланс, state machine заказа и отклика (таблицы переходов), модерация, фильтры
├── 08-integrations.md    # интерфейсы SmsProvider/PaymentProvider/PushProvider/Storage/Moderator/Maps + их dev-моки
├── 09-decisions.md       # ADR-лог: № / дата / контекст / решение / последствия
├── 10-status.md          # прогресс по майлстоунам M0–M8, что дальше, известные долги
└── glossary.md           # термины из ТЗ (заказ, отклик, подписка, каскад, автозакрытие…)
```

**Правила ведения вики (действуют всю разработку):**
1. Вики = source of truth. Расхождение кода и вики — баг: чини вики или код.
2. После каждого майлстоуна обновляй `10-status.md` и все затронутые страницы.
3. Любое отступление от этого промпта или ТЗ → запись в `09-decisions.md` (ADR), не переспрашивая пользователя по мелочам.
4. Схемы — mermaid, чтобы читались и человеком, и моделью.
5. `CLAUDE.md` в корне — короткий (≤ 60 строк): стек, команды запуска/тестов/линта, «сначала читай docs/», конвенции коммитов. Не дублируй вики — ссылайся на неё.

Первые версии страниц заполни из этого промпта (05, 06, 07 — почти готовы ниже), дальше развивай.

## 4. Дизайн-система: «Apple снаружи, Tinder/Duolingo внутри»

**Главный принцип.** Визуальный язык — iOS-минимализм, ровно как на мокапах: белый фон, один акцентный синий, тонкие бордеры, много воздуха, SF Pro. Поведение и характер — Tinder/Duolingo: свайпы, прогресс, празднования, сочные микровзаимодействия, haptics. Никакого визуального шума — вся «игра» в моушне и физике, не в декоре.

### 4.1 Токены (`lib/core/theme/tokens.dart`)

Цвета:
- `primary #2563EB`, `primaryPressed #1D4ED8`, `primarySoft #EFF4FF` (фон выбранных карточек/чипов)
- `success #22C55E` + `successSoft #E9F9EF`; `warning #F59E0B` + `warningSoft #FEF4E2`; `danger #EF4444` + `dangerSoft #FDEBEB`
- `ink #0F172A` (основной текст), `inkSecondary #64748B`, `border #E2E8F0`, `bg #FFFFFF`, `surface #F7F8FA`
- Статусные чипы: Новый → primary/primarySoft; Ожидание ответа → warning; Принят / Выполнен → success; Не выбран / Отменён → inkSecondary/surface; В работе → primary; На рассмотрении → warning.

Типографика: SF Pro (iOS) / Inter fallback (Android). LargeTitle 28/34 w700, Title 22/28 w700, Headline 17/22 w600, Body 16/22 w400, Caption 13/18 w400, Price 20/24 w700.

Геометрия: 4pt-сетка; радиусы — инпуты 12, карточки 16, CTA 14, чипы 999; CTA высота 52; инпуты 52; отступ экрана 16–20. Тени почти не используются — карточки на `border 1px` + при необходимости еле заметная тень (y2 blur8 α0.04).

### 4.2 Моушн и haptics
- Базовые длительности 160–240 мс, `Curves.easeOutCubic`; переходы страниц — Cupertino.
- Любая primary-кнопка: pressed → scale 0.97 + `HapticFeedback.lightImpact`.
- OTP: ячейка подпрыгивает при вводе (scale 1→1.08→1); ошибка — shake + heavy haptic.
- Success-экраны («Верификация пройдена», «Заказ выполнен»): зелёный чек с burst-частицами + confetti + medium haptic — как на мокапе листа 1, экран 8.
- Онбординг специалиста: толстый закруглённый progress bar в стиле Duolingo, шаги 1–4, анимированное заполнение.
- Списки: staggered появление (fade + slide 12px), pull-to-refresh — cupertino.
- Балансы/рейтинги — анимированные счётчики (implicit animation числа).

### 4.3 Tinder-механика: Order Deck (главный экран специалиста)
- Экран «Заказы» специалиста имеет три вида через segmented control: **Колода (default) / Список / Карта**.
- Колода — стопка карточек заказа: фото (или заглушка категории), чип категории, цена крупно, расстояние, описание в 2 строки, «когда удобно».
- Жесты: **свайп вправо** → bottom sheet отклика («Принять цену» / «Предложить свою»); **свайп влево** → скрыть заказ (persist в `HiddenOrder`, больше не показывать); **тап** → полная карточка заказа.
- Физика: rotation до ±8°, полупрозрачные overlay-иконки ✓/✕ растут по мере свайпа, haptic на пороге срабатывания, undo-снекбар 3 сек после скрытия.
- Пустая колода → дружелюбный empty state + кнопка «Открыть карту».

### 4.4 Duolingo-слой (delight; фичефлаг `gamification`, по умолчанию on)
- **Streak**: дни подряд с ≥ 1 откликом — чип 🔥N в профиле специалиста, мягкое празднование при +1 (без давления и дарк-паттернов).
- Крупные дружелюбные иллюстрации на pending/empty-состояниях (стиль «Проверяем ваши данные» с мокапа — простые формы, primarySoft-палитра, можно кодом/SVG).
- Празднование первого заказа / первого отклика.

### 4.5 Запрещено
Градиенты и тяжёлые тени; > 2 акцентных цветов на экране; кастомные шрифты вне SF/Inter; больше одной primary-CTA на экран; экраны-простыни (длинные формы бить на шаги с прогрессом).

## 5. Продуктовая спека — карта экранов

Формат: экран → содержимое → правила. Все UI-строки — русские (канон в Приложении A), казахский перевод в ARB с пометкой `// TODO native review`. Нумерация S-XX используется в роутере и в `docs/05-screens.md`.

### 5.1 Онбординг и auth (мокап, лист 1)
- **S-01 Welcome** — лого Zovu, слоган «Находим специалистов рядом», CTA «Начать».
- **S-02 Phone input** — заголовок «Ваш номер», префикс +7, маска `912 345-67-89`, CTA «Получить код». Валидация КЗ-номера.
- **S-03 OTP** — «Введите код из SMS», 4 ячейки, «Отправить код повторно 00:45» (таймер), TTL кода **2 минуты** (НФ-05), 5 попыток → новый код. В dev код всегда `1111` и печатается в лог API.
- **S-04 Role choice** — «Кто вы?»: карточки «Я заказчик» / «Я специалист». Показывается только новым номерам; повторный вход → сразу в активную роль.
- **S-05 Specialist form (шаг 1)** — «Расскажите о себе»: ФИО*, дата рождения* (ДД.ММ.ГГГГ), основная категория* (справочник), доп. категории (мультивыбор), «О себе» (0/500), «Загрузить диплом» (опционально, пунктирная кнопка → S-05a: jpg/png/pdf ≤ 10 МБ).
- **S-06 Verification (шаг 2)** — «Подтверждение личности»: две карточки-загрузки «Селфи» и «Селфи с документом» (камера/галерея), CTA «Отправить на проверку». До прохождения верификации отклики заблокированы (В-06).
- **S-07 Pending** — иллюстрация часов, «Проверяем ваши данные», текст «Обычно проверка занимает от нескольких минут до 24 часов. Мы пришлём уведомление». Обновление по push + polling. В dev флаг `AUTO_APPROVE_VERIFICATION=true` — одобрение через ~5 сек.
- **S-08 Success** — зелёный чек + confetti, «Верификация пройдена», CTA «Продолжить» → рабочее пространство специалиста.
- Заказчик после S-04 заполняет только ФИО и попадает сразу на S-20.

### 5.2 Специалист (мокап, лист 2)
- **S-10 Карта** — маркеры активных заказов в категориях специалиста, кластеризация при зуме, тап по маркеру → мини-карточка → S-12. Таббар специалиста: **Карта / Заказы / Отклики / Профиль**.
- **S-11 Заказы (Колода/Список/Карта)** — segmented; в списке фильтры-чипы «Новые (N)» / «Рядом» / «Все». Блок **«Новые»** — заказы младше 1 минуты, отдельно сверху (С-03), затем перетекают в общий список (С-04). Сортировка по расстоянию. Карточка: чип «Новый», название, адрес, цена, расстояние, «N мин назад».
- **S-12 Карточка заказа** — чип категории, название, цена крупно, расстояние, адрес, фото, «Описание», строки «Когда удобно» / «Длительность», CTA «Откликнуться».
- **S-13 Отклик** — сводка заказа, поле «Ваша цена» (предзаполнено ценой заказчика), строки «Комиссия сервиса» и «Вы получите» (пересчёт на лету), кнопки «Принять цену» (primary) / «Предложить свою» (secondary), подпись «Клиент увидит вашу цену и примет решение».
- **S-14 Отклики** — табы «Все / Активные / Завершённые»; карточки со статус-чипами: «Ожидание ответа», «Принят», «Не выбран».
- **S-15 Баланс** — синяя карточка: «Текущий баланс», «Следующее списание <дата> · 100 ₸/день», чип «Подписка активна/неактивна»; ниже «История операций» (пополнения +, списания −, комиссия −).
- **S-16 Пополнение** — «Введите сумму», пресеты 1000/2000/5000/10000 ₸, способ оплаты Kaspi / Банковская карта (радио), «Комиссия не взимается», CTA «Пополнить» → мок-успех → обновление баланса с анимацией.
- **S-17 Подписка неактивна** — полноэкранный блок при попытке отклика с балансом < 100: иллюстрация кошелька, «Пополните баланс», текст «Баланс нулевой, поэтому отклики недоступны…», CTA «Пополнить баланс», ссылка «Подробнее о подписке» (БП-06).
- **S-18 Профиль специалиста** — аватар, имя, ★рейтинг, «N заказов», бейдж «Дипломированный ✓» (если есть), «Подписка неактивна» метка (если так), категории (+ «Добавить» → предложение своей категории, К-02), Портфолио (N работ), Отзывы (N), О себе. Streak-чип при `gamification=on`.

### 5.3 Заказчик (мокап, лист 3)
- **S-20 Создание заказа** — первый экран заказчика после входа: категория* (шит справочника + «Предложить свою»), «Что надо сделать»*, фото до 5 (сжатие перед загрузкой, НФ-08), «Бюджет (₸)»*, «Адрес» (автоопределение по GPS + ручной ввод), CTA «Опубликовать». Иконка ⓘ → фильтры (S-21). Таббар заказчика: **Заказы / Отклики / Чаты / Профиль**.
- **S-21 Фильтры подбора** — «Только дипломированные» (свитч, Ф-02), «Мин. рейтинг» (1.0–5.0 шаг 0.5, Ф-03), «Опыт работы» (≥5 / ≥20 / ≥50 заказов, Ф-04), «Расстояние» (слайдер 1–50 км, Ф-05), «Сбросить», CTA «Показать N специалистов» (живой счётчик). «Сохранить как пресет» (Ф-09, low priority). Специалисты вне фильтров заказ не видят (Ф-07). Если 10 минут нет откликов → push «Смягчите фильтры» (Ф-08).
- **S-22 Карта заказчика** — свой заказ по центру, маркеры подходящих специалистов, снизу карточка «Ваш заказ: <название> · <бюджет> · до N км».
- **S-23 Отклики (список)** — «Получено N откликов», карточки: фото, имя, бейдж ✓, ★рейтинг (кол-во), расстояние, «Цена N ₸».
- **S-24 Деталь отклика** — профиль специалиста (бейдж «Дипломированный», о себе, ★, N заказов, расстояние), «Предложение N ₸», «Сроки», кнопки «Отклонить» / «Принять». Принятие → остальные отклики автоматически «Не выбран» + push (каскад).
- **S-25 Активный заказ** — статус-баннер «В работе · Специалист уже приступил к выполнению», данные заказа, исполнитель (тап → профиль), «Договорились <дата, время>», CTA «Завершить заказ».
- **S-26 Завершение** — баннер «Ожидает подтверждения — проверьте работу и подтвердите выполнение» → после подтверждения специалистом зелёный блок «Выполнен · Заказ успешно завершён <дата>».
- **S-27 Оценка и отзыв** — «Как прошла работа?», 5 звёзд с подписью (Плохо…Отлично), комментарий ≤ 300 (счётчик 97/300), CTA «Оставить отзыв». Обе стороны оценивают друг друга (О-04: один раз на заказ).

### 5.4 Общие (мокап, лист 4)
- **S-30 Чат** — открывается автоматически после принятия отклика (Ч-01); хедер: имя, роль, онлайн-статус, «Пожаловаться» (→ тикет поддержки с привязкой заказа, Ч-08); пузыри с временем и галочками доставки/прочтения; только текст; push о новом сообщении (Ч-05); история в «Моих заказах» (Ч-06); после завершения и оценки чат read-only (Ч-07).

## 6. Бизнес-правила (source of truth → продублируй в `docs/07-business-rules.md`)

### 6.1 Баланс и подписка (Б-*, БП-*)
1. После регистрации: `balance = 0`, подписка неактивна, новые отклики заблокированы (Б-01).
2. Cron ежедневно в 00:00 **Asia/Almaty**: для каждого специалиста с активной подпиской — списание 100 ₸ (`Transaction: subscription`). Если `balance < 100` на момент списания → подписка неактивна + push «Низкий баланс» (Б-03…Б-05).
3. Пополнение: если подписка была неактивна и после пополнения `balance ≥ 100` → немедленное списание 100 ₸ и активация, статус «Подписка активна» (БП-07).
4. При неактивной подписке: новые отклики блокируются экраном S-17 (БП-02, БП-06); **ранее отправленные отклики остаются активными** и видны заказчикам (БП-03); если такой отклик принят — специалист получает и выполняет заказ (БП-04); в профиле метка «Подписка неактивна» (БП-05).

### 6.2 Комиссия за заказ (ADR-001 — уже решено)
В ТЗ v1.2 комиссии нет, в мокапах есть («Комиссия сервиса 250 ₸», «Вы получите 4 750 ₸», транзакция «Комиссия за заказ −250 ₸»). **Реализуем по мокапам:** `ORDER_COMMISSION_PCT=5` (env, можно 0). Комиссия рассчитывается от цены принятого отклика и списывается с баланса специалиста **в момент принятия отклика заказчиком**; баланс может уйти в минус (новые отклики при `< 100` и так заблокированы). Экран S-13 показывает комиссию и «Вы получите» до отправки отклика. Занеси как ADR-001 в `09-decisions.md`.

### 6.3 State machine заказа и отклика (ЗВ-*, разделы 4–5, 9, 11 ТЗ)

Order: `active → in_progress → awaiting_confirmation → completed | completed_auto` ; из `active|in_progress` возможны `cancelled`, из любых активных — `disputed` (через тикет).

Bid: `pending → accepted | declined | not_selected`.

Правила переходов:
- Публикация → `active`; отклики от подходящих под фильтры специалистов.
- Заказчик принимает отклик → bid `accepted`, все остальные pending-отклики этого заказа → `not_selected` + push каждому; заказ → `in_progress`; создаётся чат.
- **Только заказчик** жмёт «Завершить» (ЗВ-01) → `awaiting_confirmation` + push специалисту.
- Специалист «Подтвердить выполнение» в течение 24 ч → `completed` (ЗВ-02). Не подтвердил за 24 ч → cron → `completed_auto` (ЗВ-03).
- Зеркальный кейс (ЗВ-04): специалист может отметить «Работа выполнена с моей стороны»; если заказчик 24 ч бездействует → `completed_auto`.
- Отмена: до принятия отклика — свободно; после — по согласию обеих сторон (кнопка «Предложить отмену» → подтверждение второй стороной) или через поддержку (ЗВ-07).
- Спор: любая сторона открывает тикет с привязкой заказа → флаг `disputed` («На рассмотрении») до решения админа (ЗВ-06).
- Статусы в истории заказчика: Активный / В работе / Ожидает подтверждения / Выполнен / Выполнен (автозакрытие) / Отменён / На рассмотрении (ИЗ-02). У специалиста: Отклик отправлен / Принят / Выполняется / Ожидает подтверждения / Выполнен / Не выбран / Отменён (ИС-02).

### 6.4 Оценки и модерация (О-*, ОМ-*)
- 1–5★, комментарий ≤ 300, один раз на заказ с каждой стороны (О-01…О-04); после автозакрытия окно оценки — 7 дней (ЗВ-05).
- Публикация мгновенная, но текст проходит `Moderator.check(text)` (интерфейс): dev-реализация — стоп-словарь RU/KZ (мат, оскорбления, ссылки/спам-паттерны); при срабатывании — блок с предложением переформулировать (ОМ-01, ОМ-02). Прод-адаптер `AnthropicModerator` — заглушка с TODO.
- «Пожаловаться» на отзыв: причина (оскорбление / ложь / не относится к заказу / иное) → очередь админа; отзыв остаётся видимым до решения (ОМ-03, ОМ-04). Админ скрывает или возвращает; уведомления обеим сторонам (ОМ-05). Скрытый отзыв исключается из среднего рейтинга (ОМ-06, пересчёт).
- Автор может редактировать отзыв 24 ч после публикации (ОМ-07).
- Заказчик видит рейтинг и отзывы специалиста в карточке отклика, скрытые исключены (О-05, ОМ-08).

### 6.5 Категории (К-*)
- Seed-справочник: Электрика, Сантехника, Уборка, Ремонт, Сборка мебели, Бытовая техника, Отделка, Грузоперевозки, Клининг после ремонта, Компьютерная помощь, Красота, Репетиторство.
- «Предложить свою»: специалист вводит название → статус `pending`, никому не видна (К-05) → админ одобряет/отклоняет (SLA ≤ 1 ч) + push (К-04) → одобренная доступна всем (К-06).
- Бонус за одобренную новую категорию: **3 дня подписки бесплатно** — поле `subscriptionFreeUntil = max(now, freeUntil) + 3d`, cron пропускает списание до этой даты (ADR-002).

### 6.6 Дипломированный специалист (ДС-*)
Загрузка jpg/png/pdf ≤ 10 МБ (валидация на клиенте и сервере); статусы `none → pending → approved | rejected(reason)`; SLA 48 ч; push о результате; бейдж в профиле и карточке отклика; повторная загрузка после отказа; админ может отозвать статус; файлы — в приватном бакете MinIO, доступ только админ-эндпоинтам (НФ-09).

### 6.7 Auth (НФ-05)
OTP 4 цифры, TTL 2 мин, resend через 45 с, 5 неверных попыток → код сгорает. JWT access/refresh, refresh-ротация. Один аккаунт = один номер; роли — флаги на пользователе.

## 12. Definition of Done

- `docs/` актуален, `openapi.json` свежий, README поднимает проект за ≤ 5 команд.
- `flutter analyze` чисто, jest зелёный.
